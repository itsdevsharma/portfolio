1. Overview

This document describes the CI/CD pipeline set up for the Portfolio2 project using GitHub Actions for automation and Docker for containerization. The pipelines ensure that every code change is automatically validated and, upon merging to main, deployed to the production server with zero manual steps.

TermWhat it means
CI — Continuous IntegrationAutomatically test and build the code on every push or pull request
CD — Continuous DeploymentAutomatically build Docker images and deploy to the server when CI passes
GitHub ActionsGitHub's built-in automation system that runs workflows defined in .yml files
DockerPackages the app into a container — a lightweight, portable unit that runs the same everywhere
GHCRGitHub Container Registry — free image hosting built into GitHub, no extra account needed
Docker ComposeRuns multiple containers (frontend + backend) together with one command

3. CI Pipeline — Continuous Integration

File: .github/workflows/ci.yml

Trigger Conditions

on:
  push:
    branches: [main]        # runs when you push directly to main
  pull_request:
    branches: [main]        # runs when a PR targets main — blocks merge if it fails

Job 1: Frontend — Lint, Test & Build

All steps run inside the frontend/ directory.

StepCommandPurpose
Checkoutactions/checkout@v4Downloads your repository code into the runner
Setup Nodeactions/setup-node@v4Installs Node.js 20 with npm cache enabled (faster installs)
Installnpm ciClean install from lock file — deterministic, reproducible
Lintnpm run lintESLint checks for code quality issues and anti-patterns
Testnpm testRuns Vitest unit tests — fails the pipeline if any test fails
Buildnpm run buildVite compiles the React app to static files in dist/
Upload artifactupload-artifact@v4Saves the built dist/ folder for 1 day for inspection
Why npm ci instead of npm install?
npm ci deletes node_modules and installs exactly what's in package-lock.json. It never modifies the lock file, ensuring every pipeline run gets identical dependencies.

Job 2: Backend — Install & Validate

All steps run inside the backend/ directory.

StepCommandPurpose
Checkoutactions/checkout@v4Downloads repository code
Setup Nodeactions/setup-node@v4Node.js 20 with npm cache
Installnpm ciClean install of backend dependencies
Syntax checknode --check index.jsParses all JS for syntax errors without running the server (no database required)
Why not run the backend server in CI?
The backend connects to MongoDB on startup. Starting the full server in CI would require a real or mock database. node --check validates syntax without any external dependencies.

4. CD Pipeline — Continuous Deployment

File: .github/workflows/cd.yml

Trigger Condition

on:
  push:
    branches: [main]        # only on direct push to main (not PRs)

Permissions

permissions:
  contents: read
  packages: write           # required to push images to GitHub Container Registry

Job 1: Build & Push Docker Images

Step 1 — Set up Docker Buildx

uses: docker/setup-buildx-action@v3

Buildx is the modern Docker build engine (BuildKit). It enables:

  • Layer-by-layer caching — only rebuilds what changed
  • Parallel build stages
  • Multi-platform image support (e.g., linux/amd64 and linux/arm64)

Step 2 — Login to GitHub Container Registry

uses: docker/login-action@v3
with:
  registry: ghcr.io
  username: ${{ github.actor }}       # your GitHub username, auto-injected
  password: ${{ secrets.GITHUB_TOKEN }} # auto-generated token, no setup needed

GHCR (ghcr.io) is GitHub's free container registry. Images are stored alongside your repository. GITHUB_TOKEN is automatically available in every GitHub Actions workflow — no manual secret needed.

Step 3 — Build & Push Backend Image

uses: docker/build-push-action@v6
with:
  context: ./backend                         # uses backend/Dockerfile
  push: true
  tags: |
    ghcr.io/yourname/portfolio2-backend:latest        # always updated
    ghcr.io/yourname/portfolio2-backend:abc1234       # pinned to this exact commit
  cache-from: type=gha
  cache-to: type=gha,mode=max

Two tags explained:

TagPurpose
:latestAlways points to the newest build — used by the server when pulling
:<git-sha>Pinned to this exact commit — used for rollback if something breaks

Build cache: type=gha stores Docker layer cache in GitHub Actions cache storage. On the second build, only layers that changed are rebuilt — dramatically faster builds.

Step 4 — Build & Push Frontend Image

uses: docker/build-push-action@v6
with:
  context: ./frontend
  build-args: |
    VITE_API_URL=${{ secrets.VITE_API_URL }}   # production backend URL baked into the bundle
Why is the API URL a build argument?
Vite compiles environment variables (VITE_*) into the JavaScript bundle at build time. Unlike a backend where you can read env vars at runtime, a React SPA has no server — so the URL must be embedded during the build. That's why it's passed as a Docker build argument.

Job 2: Deploy to Server

needs: build-and-push   # only runs if Job 1 succeeded

Step 1 — Copy docker-compose.yml to server (SCP)

uses: appleboy/scp-action@v0.1.7
with:
  source: docker-compose.yml
  target: ~/portfolio2

SCP (Secure Copy Protocol) transfers the docker-compose.yml file over SSH to your server. This ensures the server always has the latest compose configuration.

Step 2 — Pull & Restart Containers (SSH)

uses: appleboy/ssh-action@v1
script: |
  cd ~/portfolio2
  echo "TOKEN" | docker login ghcr.io --password-stdin   # authenticate to pull private images
  export VITE_API_URL=https://api.yourdomain.com
  docker compose pull                                      # download latest images from GHCR
  docker compose up -d --remove-orphans                   # restart in background, clean up old
  docker image prune -f                                    # delete dangling old images
CommandWhat it does
docker compose pullDownloads the newly pushed :latest images
docker compose up -dStarts containers in detached (background) mode
--remove-orphansRemoves containers for services removed from the compose file
docker image prune -fDeletes old image versions to free disk space

5. Docker Compose

File: docker-compose.yml

Docker Compose is used both for local development (docker compose up --build) and production (pulling pre-built images from GHCR).

services:
  backend:
    build:
      context: ./backend        # build image from backend/Dockerfile
    container_name: portfolio-backend
    restart: unless-stopped     # auto-restart on crash or server reboot
    env_file: ./backend/.env    # loads MONGO_URI, RESEND_API_KEY, PORT, etc.
    ports:
      - "5000:5000"             # host port : container port
    volumes:
      - backend_uploads:/app/uploads   # persists uploaded resume file across restarts
    networks:
      - portfolio-net

  frontend:
    build:
      context: ./frontend
      args:
        VITE_API_URL: ${VITE_API_URL:-http://localhost:5000}
        # ↑ uses $VITE_API_URL env var if set, falls back to localhost for local dev
    container_name: portfolio-frontend
    restart: unless-stopped
    ports:
      - "80:80"                 # serves on port 80 (standard HTTP)
    depends_on:
      - backend                 # Docker starts backend first, then frontend
    networks:
      - portfolio-net

volumes:
  backend_uploads:              # named volume — survives container deletion/rebuild

networks:
  portfolio-net:
    driver: bridge              # isolated virtual network

Key Concepts

ConceptExplanation
restart: unless-stoppedContainer auto-restarts on crash and on server reboot, but not if you manually stop it
env_fileReads all key=value pairs from .env file — keeps secrets out of the compose file
volumes: backend_uploadsNamed volume persists the uploaded resume PDF even when the container is rebuilt or deleted
depends_onDocker starts the backend container before the frontend — prevents frontend from starting with no API available
portfolio-netAn isolated virtual network — containers can talk to each other by service name (e.g., http://backend:5000)
:- default syntax${VITE_API_URL:-http://localhost:5000} means: use the env var if set, otherwise use the fallback

6. Dockerfiles

Frontend Dockerfile — Multi-Stage Build

File: frontend/Dockerfile

# ── Stage 1: Build ───────────────────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

ARG VITE_API_URL=http://localhost:5000    # receives value from docker compose / CI
ENV VITE_API_URL=$VITE_API_URL

COPY package*.json ./
RUN npm ci                                # install deps first (Docker cache layer)

COPY . .
RUN npm run build                         # produces dist/ folder

# ── Stage 2: Serve ───────────────────────────────────────────────
FROM nginx:alpine                         # tiny final image (no Node.js, no source code)

COPY --from=builder /app/dist /usr/share/nginx/html   # copy only the built files
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Why multi-stage? The builder stage uses a full Node.js image (~180MB) with all dev tools. The final image only contains nginx + the compiled HTML/JS/CSS (~25MB). The source code, node_modules, and build tools never reach production.

nginx.conf — SPA Routing

location / {
    try_files $uri $uri/ /index.html;    # all unknown routes → index.html
}

Without this, refreshing the page on a React route (e.g., /projects) would return a 404 from nginx because the file doesn't exist on disk — React Router handles the routing in the browser.

Backend Dockerfile

File: backend/Dockerfile

FROM node:20-alpine

WORKDIR /app

COPY package*.json ./
RUN npm ci --omit=dev          # installs only production deps (no devDependencies)

COPY . .
RUN mkdir -p uploads            # ensure uploads directory exists

EXPOSE 5000
CMD ["node", "index.js"]

--omit=dev skips devDependencies (like nodemon, test frameworks) — smaller, more secure image.

7. GitHub Secrets Setup

Go to your repository → Settings → Secrets and variables → Actions → New repository secret

Secret NameValueRequired for
VITE_API_URLYour production backend URL
e.g. https://api.yourdomain.com		Frontend build — baked into the JS bundle
SSH_HOSTYour server's IP address or hostnameCD deploy step
SSH_USERSSH login username (e.g. ubuntu, root)CD deploy step
SSH_PRIVATE_KEYContents of your ~/.ssh/id_rsa private keyCD deploy step
GITHUB_TOKENAutomatically available — no setup neededPushing images to GHCR
Never commit secrets to the repository. All sensitive values (API keys, database URIs, SSH keys) must go in GitHub Secrets — not in .env files that are committed to git.

8. End-to-End Flow

#EventWhat happens
1Developer pushes code to mainGitHub Actions triggers both CI and CD workflows
2CI — Frontend jobLint → test → build. If any step fails, the workflow is marked as failed
3CI — Backend jobInstall → syntax check. Runs in parallel with frontend job
4CD — Build & Push (Job 1)Builds Docker images for frontend and backend, pushes to GHCR with :latest and :sha tags
5CD — Deploy (Job 2)Copies docker-compose.yml to server via SCP, then SSHs in and runs docker compose pull && up -d
6ProductionNew containers are running. Old ones are stopped. Disk is cleaned up.

9. Running Locally with Docker

Start everything

cd portfolio2
docker compose up --build
  • Frontend → http://localhost (port 80)
  • Backend API → http://localhost:5000

Stop containers

docker compose down

Stop and delete volumes (reset uploads)

docker compose down -v

View logs

docker compose logs -f             # all services
docker compose logs -f backend     # backend only

Rebuild a single service after code changes

docker compose up --build frontend

Roll back to a previous version on the server

# SSH into your server, then:
docker pull ghcr.io/yourname/portfolio2-backend:<old-sha>
docker pull ghcr.io/yourname/portfolio2-frontend:<old-sha>
docker tag ghcr.io/yourname/portfolio2-backend:<old-sha> ghcr.io/yourname/portfolio2-backend:latest
docker tag ghcr.io/yourname/portfolio2-frontend:<old-sha> ghcr.io/yourname/portfolio2-frontend:latest
docker compose up -d
